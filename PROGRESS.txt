  ARCHITECTURE OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Clean Architecture, 4 layers:
  - ClinicFlow.Domain         — Entities, enums, repository interfaces
  - ClinicFlow.Application    — CQRS via MediatR, FluentValidation, DTOs
  - ClinicFlow.Infrastructure — EF Core DbContext, repositories, migrations
  - ClinicFlow (API)          — Controllers, middleware, Program.cs

Pattern: every feature lives at ClinicFlow.Application/Features/<Domain>/
  Commands/Create<X>/   — Command + Handler + Validator
  Commands/Update<X>/   — Command + Handler + Validator
  Queries/Get<X>s/      — Query + Handler (+ optional Validator)
  DTOs/                 — <X>Dto, <X>SummaryDto

Multi-tenancy: every entity has ClinicId (Guid). All queries filter by it.
  TenantMiddleware resolves ClinicId from JWT → Staff table → HttpContext.Items["ClinicId"]
  Dev bypass: DevClinicId in appsettings.Development.json (hardcoded GUID, no JWT needed)

Error handling: ExceptionHandlingMiddleware translates exceptions → HTTP codes:
  ValidationException         → 400
  NotFoundException           → 404
  ConflictException           → 409
  UnauthorizedAccessException → 401
  anything else               → 500

Database: SQL Server LocalDB
  Connection: Server=(localdb)\mssqllocaldb;Database=ClinicFlowDb

EF migration commands (must use full csproj paths):
  dotnet ef migrations add <Name> --project ClinicFlow.Infrastructure/ClinicFlow.Infrastructure.csproj --startup-project ClinicFlow.csproj
  dotnet ef database update       --project ClinicFlow.Infrastructure/ClinicFlow.Infrastructure.csproj --startup-project ClinicFlow.csproj

Dev run command:
  dotnet run --project ClinicFlow.csproj --urls "http://localhost:5062"

  DOMAIN ENTITIES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All entities follow the same pattern:
  - Factory method Create(...) — no public constructor
  - Mutation methods Update(...), Deactivate() — no public setters
  - BaseEntity provides: Id (Guid), CreatedAt (DateTime), UpdatedAt (DateTime)

1. CLINIC
   File: ClinicFlow.Domain/Entities/Clinic.cs
   Properties: Name, Phone, Address, LogoUrl?, WorkingHoursStart (TimeOnly),
               WorkingHoursEnd (TimeOnly), SlotDurationMinutes, IsActive
   Methods: Create(...), Update(...), Deactivate(), Activate()
   Note: No ClinicId — Clinic IS the top-level tenant root.

2. STAFF
   File: ClinicFlow.Domain/Entities/Staff.cs
   Properties: ClinicId, Name, Email, Role (StaffRole enum), IsActive
   Methods: Create(clinicId, name, email, role), Update(name, role), Deactivate()
   Enums: StaffRole { Owner=0, Receptionist=1 }

3. CUSTOMER
   File: ClinicFlow.Domain/Entities/Customer.cs
   Properties: ClinicId, Name, Phone, Email?, DateOfBirth?, Gender?, Notes?
   Methods: Create(clinicId, name, phone, email?, dob?, gender?, notes?), Update(...)
   Enums: Gender { Male=0, Female=1, Other=2 }

4. TREATMENT
   File: ClinicFlow.Domain/Entities/Treatment.cs
   Properties: ClinicId, Name, Description?, DurationMinutes, FollowUpIntervalDays (int?),
               Price (decimal?), IsActive
   Methods: Create(...), Update(...), Deactivate()
   Note: FollowUpIntervalDays drives the follow-up suggestion calculation.

5. APPOINTMENT
   File: ClinicFlow.Domain/Entities/Appointment.cs
   Properties: ClinicId, CustomerId, Date (DateOnly), TimeSlot (TimeOnly),
               Status (AppointmentStatus), Notes?
   Navigation: Customer, Treatments (List<AppointmentTreatment>)
   Methods: Create(...), Update(date, timeSlot, notes), Cancel(), Complete(), MarkNoShow()
   Enums: AppointmentStatus { Scheduled=0, Completed=1, Cancelled=2, NoShow=3 }

6. APPOINTMENTTREATMENT  (join entity)
   File: ClinicFlow.Domain/Entities/AppointmentTreatment.cs
   Properties: AppointmentId, TreatmentId, Notes?
   Navigation: Treatment
   Methods: Create(appointmentId, treatmentId, notes?)
   Role: bridges Appointment ↔ Treatment (one appointment → many treatments)

  CLINICS   /api/clinics
────────────────────────────────────────────────────────────────────────────────

NOTE: Unlike every other controller, Clinics endpoints are NOT tenant-scoped —
Clinic IS the tenant root, so GET/PUT/PATCH-by-id operate across ALL clinics
(admin-style). Only GET /me, POST, and PUT /me are scoped to the caller's own
clinic via ICurrentClinicService. This supports a "Clinics" admin list page
in the frontend (see clinicflow-web/PROGRESS.txt).

GET /api/clinics
  Description : Returns ALL clinics (not tenant-scoped — admin/list view).
  Auth        : Owner only (once role auth is wired).
  Response 200: [ <ClinicDto>, ... ]  (same shape as GET /me response below,
                each including "isActive")

GET /api/clinics/me
  Description : Returns the current clinic's full profile.
  Auth        : Any authenticated staff member.
  Response 200:
    {
      "id": "guid",
      "name": "string",
      "phone": "string",
      "address": "string",
      "logoUrl": "string | null",
      "workingHoursStart": "08:00:00",
      "workingHoursEnd": "18:00:00",
      "slotDurationMinutes": 30,
      "isActive": true,
      "createdAt": "datetime"
    }
  Errors: 404 if no clinic exists yet for this tenant.

POST /api/clinics
  Description : Creates a new clinic (first-time setup).
  Auth        : Owner only (once role auth is wired).
  Body:
    {
      "name": "string (required)",
      "phone": "string (required)",
      "address": "string (required)",
      "logoUrl": "string | null",
      "workingHoursStart": "08:00:00",
      "workingHoursEnd": "18:00:00",
      "slotDurationMinutes": 30
    }
  Response 201: { "id": "guid" }
  Errors: 400 on validation failure, 409 if name already exists.

PUT /api/clinics/me
  Description : Updates the current clinic's profile.
  Auth        : Owner only.
  Body        : Same shape as POST body above.
  Response 204.
  Errors: 400 on validation, 404 if clinic not found.

PUT /api/clinics/{id}
  Description : Updates ANY clinic's profile by id (admin edit, not tenant-scoped).
                Command: UpdateClinicByIdCommand. Route id overwrites any Id in
                the request body via `command with { Id = id }` (matches the
                Staff/Treatments update-by-id convention).
  Auth        : Owner only (once role auth is wired).
  Body        : Same shape as POST body above.
  Response 204.
  Errors: 400 on validation, 404 if not found, 409 on name conflict with another clinic.

PATCH /api/clinics/{id}/deactivate
  Description : Sets IsActive = false on the given clinic.
  Auth        : Owner only.
  Body        : None.
  Response 204.
  Errors: 404 if not found.

PATCH /api/clinics/{id}/activate
  Description : Sets IsActive = true on the given clinic (reactivation).
  Auth        : Owner only.
  Body        : None.
  Response 204.
  Errors: 404 if not found.

POST /api/appointments
  Description : Creates a new appointment and links one or more treatments to it.
                Validates that the time slot is not already taken (skips cancelled).
  Auth        : Owner, Receptionist.
  Body:
    {
      "customerId": "guid (required)",
      "date": "2026-07-15",
      "timeSlot": "10:00:00",
      "notes": "string | null",
      "treatments": [
        {
          "treatmentId": "guid",
          "notes": "string | null"
        }
      ]
    }
  Response 201: { "id": "guid" }
  Errors: 400 on validation, 404 if customer/treatment not found,
          409 if time slot is already booked.

PUT /api/appointments/{id}
  Description : Updates date, time slot, notes, and treatment list for an appointment.
                Replaces the treatment list entirely — send the full desired list.
                Only Scheduled appointments can be updated.
  Auth        : Owner, Receptionist.
  Body        : Same shape as POST body above.
  Response 204.
  Errors: 400, 404, 409 (slot conflict or wrong status).
  Note: Uses explicit RemoveRange + AddRange on AppointmentTreatments to avoid
        EF concurrency errors on collection updates.

PATCH /api/appointments/{id}/cancel
  Description : Cancels a Scheduled appointment. Cancelled slots become
                available for new bookings.
  Auth        : Owner, Receptionist.
  Body        : None.
  Response 204.
  Errors: 404, 409 if appointment is not in Scheduled status.

PATCH /api/appointments/{id}/complete
  Description : Marks a Scheduled appointment as Completed.
                After this, use GET /{id}/follow-up-suggestions to see
                which treatments recommend a follow-up visit.
  Auth        : Owner, Receptionist.
  Body        : None.
  Response 204.
  Errors: 404, 409 if appointment is not in Scheduled status.

PATCH /api/appointments/{id}/no-show
  Description : Marks a Scheduled appointment as NoShow (customer did not attend).
  Auth        : Owner, Receptionist.
  Body        : None.
  Response 204.
  Errors: 404, 409 if appointment is not in Scheduled status.

GET /api/appointments/{id}/follow-up-suggestions
  Description : For a Completed appointment, returns one suggestion per treatment
                that has a FollowUpIntervalDays value set. The suggested date is
                appointment.Date + treatment.FollowUpIntervalDays.
                Use the result to pre-fill a new POST /api/appointments call.
  Auth        : Owner, Receptionist.
  Response 200:
    [
      {
        "treatmentId": "guid",
        "treatmentName": "Laser Treatment",
        "followUpIntervalDays": 30,
        "suggestedDate": "2026-08-14"
      }
    ]
  Note: Returns an empty array if no treatments in the appointment have
        a follow-up interval configured.
  Errors: 404 if appointment not found,
          409 if appointment is not in Completed status.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. EF CORE COLLECTION UPDATES
   When updating an entity with a collection navigation property (like
   Appointment.Treatments), NEVER call context.Update(entity). This traverses
   the full entity graph and marks child entities as Modified. For new children
   (non-default Guid keys), EF will try to UPDATE non-existent rows → concurrency
   exception. Instead, use explicit RemoveRange + AddRange on the DbSet.

2. NAMESPACE SHADOWING — Staff entity
   The namespace ClinicFlow.Application.Features.Staff shadows the entity class
   Staff in CreateStaffCommandHandler. Fix: use the global:: prefix:
     global::ClinicFlow.Domain.Entities.Staff.Create(...)

3. SCOPED SERVICES IN MIDDLEWARE
   TenantMiddleware receives IStaffRepository via InvokeAsync parameter injection
   (NOT constructor). Constructor injection in middleware creates a singleton-scoped
   instance that breaks with scoped services.

4. STAFF REPOSITORY — GetByEmailAsync
   IStaffRepository.GetByEmailAsync has NO clinicId parameter. This is intentional:
   the middleware does a global lookup by email to identify which clinic the user
   belongs to. The per-clinic EmailExistsAsync method still has clinicId.

5. HTTPS IN DEVELOPMENT
   UseHttpsRedirection is skipped in Development (wrapped in if check in Program.cs).
   The API runs on http://localhost:5062 in dev. Do not use https:// when testing.

6. DOTNET RUN vs DOTNET EXEC
   When rebuilding after code changes: stop the running process first (the DLL gets
   locked). Use: Stop-Process -Name "dotnet" -Force
   Then rebuild: dotnet build ClinicFlow.slnx
   Then start: dotnet run --project ClinicFlow.csproj --urls "http://localhost:5062" --no-build

STEP 1 — Register the app in Entra ID (Azure AD)
  1. Go to portal.azure.com → Microsoft Entra ID → App registrations → New registration
  2. Fill in:
       Name                   : ClinicFlow (or any descriptive name)
       Supported account types: Accounts in this organizational directory only
                                (single-tenant — recommended for a clinic SaaS)
       Redirect URI           : leave blank (API only, no browser login needed)
  3. Click Register

STEP 2 — Collect the values you will need
  From the app's Overview page, copy:
    Application (client) ID  → this is your ClientId
    Directory (tenant) ID    → this is your TenantId
  Keep these handy for the next step.

STEP 3 — Expose an API scope
  1. Go to Expose an API → Add a scope
  2. Accept the suggested App ID URI: api://<ClientId>
  3. Scope name   : access_as_user
  4. Who can consent: Admins and users
  5. Click Add scope
  Note: The scope name "access_as_user" already matches what is in appsettings.json.

STEP 4 — Create at least one test user
  Go to Microsoft Entra ID → Users → New user.
  The UPN/email you set here must exactly match the Email column on a Staff row
  in the database. That email is the bridge between the Azure AD identity and
  your clinic's role system.

STEP 5 — Populate appsettings.json with real values

STEP 6 — Ensure the Staff table has a matching record
  The TenantMiddleware calls staffRepository.GetByEmailAsync(email) where
  email comes from the "preferred_username" claim in the Azure AD token.
  Before testing, make sure the database has at least one Staff row where:
    Email    = the Azure AD user's UPN (e.g. aya@yourtenant.onmicrosoft.com)
    Role     = 0  (Owner)
    IsActive = 1  (true)
  ClinicId must also reference a valid row in the Clinics table.

STEP 7 — Get a test token via Postman
  1. New Request → Authorization tab → Type: OAuth 2.0
  2. Grant type : Authorization Code
                  (or Device Authorization if no browser redirect is configured)
  3. Auth URL   : https://login.microsoftonline.com/<TenantId>/oauth2/v2.0/authorize
  4. Token URL  : https://login.microsoftonline.com/<TenantId>/oauth2/v2.0/token
  5. Scope      : api://<ClientId>/access_as_user
  6. Client ID  : <ClientId>
  7. Click Get New Access Token → log in with the test user → copy the token
  Use the token as a Bearer header on all API calls:
    Authorization: Bearer <token>
